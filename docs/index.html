Agentic Governance:
A Pre-Authorization Model for
Policy-Compliant AI Agent Calls

Language Python 3.10+
Tests 80 passing
Source github.com/sanjdow/agentic_governance

Current approaches to governing agentic AI systems conflate identity authorization with query authorization. A token proves who is calling. It says nothing about whether the specific query being executed — generated at runtime by a language model — is policy-compliant against the data it touches.

This repository is a Python reference implementation of a distinct architectural model in which governance policy originates exclusively in the data catalog, is enforced at the orchestration layer through policy-driven agent selection, and is carried into every data source call as a cryptographically signed, query-bound proof of prior authorization. The MCP server becomes a pure enforcement point, never a decision point.

A comprehensive analysis of the current vendor landscape demonstrates that no single product satisfies this architecture end-to-end. The gap is always at L3 (query proof) and L4 (MCP enforcement) — both require custom build regardless of catalog vendor.

§1

Thesis

The fundamental error in today's agentic governance architectures is that they grant access to a source, not to a query. A token proves identity; it does not prove that the specific query being executed is policy-compliant. True agentic governance requires that every tool call to a data source carries a pre-validated, signed proof that the query itself — not merely the agent — has been authorized against the governing data catalog.

The distinguishing characteristic of the model described here is its reversal of the prevailing trust assumption. In today's token-based models, the burden of proof is reactive: a token grants entry, and any subsequent query is presumed acceptable unless the receiving system detects a violation.

In this model, the burden of proof is proactive: no query reaches a data source unless it arrives with a signed attestation that it has already been found compliant. The query is the proof. The MCP server does not need to understand policy semantics. It needs only to verify a signature.

§2

Architecture

Four ordered layers, each with a single responsibility. No layer can be bypassed. Every request resolves a UserContext at §0, which propagates unchanged through L1–L4 as the root of all trust decisions.

§0 Auth
Entra ID → UserContext

Bearer token validated against JWKS. The Entra OID becomes user_id — immutable, unlike UPN. Group memberships map to data scope. App roles map to clearance level. OBO degradation applied explicitly when the token is a delegation chain token.

RS256 JWT validation group → scope role → clearance OBO constraints Managed Identity rejection
L1 Catalog
Data Catalog — sole source of governance truth

Governance rules must not be defined in gateways, orchestrators, or MCP servers. They must be derived from the data catalog — the system of record for data classifications, sensitivity levels, access rights, and regulatory constraints. Any policy change propagates from the catalog outward. No downstream component holds an independent copy of policy logic.

sensitivity classifications access rights PII column lists consent state policy versioning
L2 Orch.
Orchestrator — policy at the agent selection layer

Before any agent is invoked, the orchestrator resolves which agents are eligible given the current user context, session, and request classification — as derived from catalog policies. Non-eligible agents are never activated. This is not filtering after the fact; it is exclusion before invocation.

clearance ceiling check access right validation source allowlist scope intersection
L3 Proof
Policy Resolver — authorization must precede the query

An agent expresses intent. A Policy Resolver translates that intent into a concrete query, validates it against catalog-derived access rights, and issues an Authorized Query Proof: a cryptographically signed, short-lived document binding the specific query to the user identity, agent identity, session, and policy version at the time of issuance. The query cannot be substituted. The proof cannot be reused.

SHA-256 query binding RS256 JWT signing 120s TTL non-delegable single-use
L4 MCP
MCP server — enforcement point, not decision point

The MCP server does not evaluate whether a query is permissible. That decision has already been made and encoded in the proof. The MCP server verifies the proof's signature, checks its expiry, and either pushes catalog-derived filters down to the data source or executes the pre-validated query unchanged. Policy logic does not live in the MCP server. It never should.

signature verification replay protection filter push-down column masking
Cross-cutting controls (not a layer — applied throughout)
Context governance

Classifies inter-agent context. Redacts content above the receiving agent's clearance ceiling before passing forward.

Governed cache

Identity-scoped Redis keys, policy-version TTLs, Fernet encryption at rest. Prevents cross-identity and stale-policy cache hits.

Session isolation

Agents declare exactly which fields they need. No raw context forwarding. Need-to-know enforced at every agent hop.

§3

Failure modes addressed

Each of these is a concrete, exploitable gap in standard agentic deployments, not a theoretical concern. The framework addresses all ten with a specific implementation.

# Failure Module
01 Tokens prove identity, not query compliance. A Bearer token says who is calling — nothing about whether the SQL is permitted against the data it touches. policy_resolver/
02 RBAC cannot govern non-deterministic queries. LLM agents generate queries at runtime; the same prompt produces different surfaces. RBAC cannot govern what it cannot predict. orchestrator/
03 Cross-call aggregation. An agent chains individually permitted calls across data sources, correlating results in its context window into inferences no single source would disclose. orchestrator/eligibility.py
04 Prompt injection hijacks tool calls. Malicious instructions in retrieved data redirect the agent. The agent calls downstream services under its own permitted identity — RBAC is blind. injection_detection/
05 Context leaks across agent chains. Data retrieved under governed conditions exits the governance perimeter the moment it enters a context window. No policy follows it forward. context_governance/
06 Redis cache carries no classification metadata. Cache keys rarely incorporate identity, enabling cross-user and cross-agent data leakage through cache hits. cache/governed_cache.py
07 Context window as shared memory. Without session isolation, Agent A's full context becomes Agent B's context. Agents accumulate sensitive data outside any governed store. agents/session_isolation.py
08 MCP server as policy decision point. When the MCP server makes policy decisions, policy logic must be replicated everywhere. It drifts, becomes inconsistent, and cannot scale. mcp_server/
09 Entra ID proves who, not what. A valid Bearer token proves authentication, not that the query is policy-compliant for the data it touches. auth/
10 OBO delegation silently drops policy context. On-Behalf-Of propagates OID but loses app role assignments, data scope, and clearance level. Downstream agents have an invisible degraded context. auth/claim_mapper.py
§4

The Authorized Query Proof

The core artifact of the pre-authorization model. A cryptographically signed JWT that binds a specific query to a specific user, agent, session, and catalog policy version. Five properties make it tamper-evident and non-reusable.

JWT payload schema issued by Policy Resolver · RS256 · 120s TTL
jti
Unique proof identifier. Tracked by MCP server for replay protection — a proof can only be used once.
sub
Entra OID of the invoking user. Immutable identity anchor — survives OBO delegation when embedded in the proof.
agent_id
The agent the proof was issued for. A different agent presenting this proof is rejected — proof is non-delegable.
query_hash
SHA-256 of the canonicalized query. The query cannot be substituted at the MCP layer — any deviation is detected.
policy_version
Catalog policy version at time of issuance. Proofs issued under superseded policy versions are detectable.
allowed_filters
Catalog-derived row filters and column masks. Pushed down to the data source by the MCP enforcement server.
exp
Expiry timestamp. Default 120 seconds from issuance. No refresh mechanism — expired proofs are cryptographically unextendable.
# Requesting a proof
proof = resolver.request_proof(
    session=session,        # contains UserContext with Entra OID
    agent=retrieval_agent,
    query="SELECT model, rate FROM quality.defect_rates",
    asset_ids=["division_quality_metrics"],
)

# The proof is bound to this exact query string
# Any modification to the query will be detected at L4
assert proof.matches_query("SELECT model, rate FROM quality.defect_rates")

# Governed MCP call — proof travels with the query
result = mcp.handle_tool_call(MCPToolCall(
    tool="query_datasource",
    arguments={"source": "snowflake", "query": query, "agent_id": "retrieval_agent"},
    proof=proof,
))
§5

Entra ID integration

Entra ID provides the identity anchor. The Authorized Query Proof provides the authorization artifact. These solve different problems. The auth/ module bridges them by translating Entra claims into the governance framework's UserContext.

What Entra ID provides
Principal authentication

Users, Managed Identities, and Service Principals authenticated via OAuth 2.0/OIDC. The OID is the immutable identity anchor — unlike UPN, it does not change on username renames.

Conditional Access pre-filter

Location, device compliance, and MFA enforcement can gate token issuance — a coarse L2-style pre-filter before agents are invoked.

Group and role claims

Group OIDs map to data scope values; app role values map to clearance levels via configurable tables in EntraConfig.

What Entra ID cannot prove
Query-level compliance

A valid Bearer token proves authentication — not that the specific query being executed is policy-compliant for the asset it touches.

Agent vs. user identity

When an LLM agent calls an MCP server using Managed Identity, the token belongs to the agent's service principal. The invoking user is invisible to the data source.

OBO delegation integrity

On-Behalf-Of propagates OID and UPN — but not app role assignments. Data scope and clearance level are silently lost in delegation chains.

The remedy: embed the Entra OID in the Authorized Query Proof as sub. Identity, data scope, clearance, and query hash travel together in a single signed artifact. Even when downstream agents never see the original Bearer token, the identity and policy context are preserved and verifiable.

The apply_obo_constraints() method in auth/claim_mapper.py explicitly models the policy context loss that occurs in OBO chains — clearing brand_scope and reducing clearance_level to INTERNAL. This makes the degradation visible rather than allowing downstream agents to silently operate with stale context.

§6

Module overview

auth/ Entra ID authentication. Token validation against JWKS, claim mapping (groups → scope, roles → clearance), OBO constraint modelling, test token factory.
catalog/ L1 data catalog. In-memory stub with runtime-queryable APIs for sensitivity classifications, access rights, consent state, row filters, and policy versioning. Replace with a real catalog in production.
policy_resolver/ L3 Policy Resolver. Issues RS256-signed Authorized Query Proofs. Handles revocation and session-level invalidation. Private key never leaves this module.
orchestrator/ L2 agent eligibility. Resolves eligible agents against catalog policy before invocation. Blocks agents where the user's clearance is below the agent's ceiling.
mcp_server/ L4 MCP Governance Server. Pure enforcement point. Verifies signature, expiry, and replay. Routes to filter push-down or pass-through. Never decides policy.
context_governance/ Inter-agent context middleware. Heuristic PII and sensitivity classifier. Intercepts context between agent hops. Redacts content above the receiving agent's clearance ceiling with overlapping chunk windows to prevent boundary evasion.
injection_detection/ 4-layer prompt injection detector. Critical patterns, boundary analysis, tool-redirect detection, delimiter injection scanning. Applied before any retrieved content reaches an agent's context.
cache/ Governed Redis wrapper. Identity-scoped keys (user + agent + query hash). Policy-version-bound TTLs. Fernet encryption at rest. Falls back to in-process store when Redis is unavailable.
agents/ Session isolation. SessionStateStore with need-to-know ContextRequest. Agents declare exactly which output types and field names they require — no raw context forwarding.
core/ Shared types. All models, exceptions, and identifier validation. SensitivityLevel comparisons. AuthorizedQueryProof schema. UserContext and AgentContext with SQL-injection-safe identifier enforcement.

Installation

Requires Python 3.10+. No external services or Azure credentials required to run the full test suite.

# Clone
git clone https://github.com/sanjdow/agentic_governance.git
cd agentic_governance

# Virtual environment
python3 -m venv .venv && source .venv/bin/activate

# Install (requirements.txt or editable)
pip install -r requirements.txt
# — or —
pip install -e ".[test]"           # adds pytest + coverage
pip install -e ".[redis]"          # real Redis backend
pip install -e ".[all]"            # everything

# Verify — 80 tests should pass
pytest tests/ -v

# Core governance demo (all 11 controls)
python examples/full_scenario.py

# Entra ID auth demo (no Azure credentials needed)
python examples/demo_entra_auth.py

Optional: real Redis

import redis
from cache.governed_cache import GovernedCache
from catalog.catalog import build_demo_catalog

catalog = build_demo_catalog()
cache = GovernedCache(
    catalog=catalog,
    redis_client=redis.Redis(host="localhost", port=6379, db=0),
    encrypt=True,
)

Optional: Entra ID production configuration

# Required environment variables
export ENTRA_TENANT_ID="xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
export ENTRA_CLIENT_ID="xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
export ENTRA_AUDIENCE="api://your-client-id"

# Group → data scope mapping
export ENTRA_BRAND_GROUP_MAP='{"group-oid-analysts": ["scope_a"]}'

# App role → clearance mapping
export ENTRA_CLEARANCE_ROLE_MAP='{"DataAnalyst":"confidential","DataAdmin":"restricted"}'

For development and testing, none of the above variables are required. EntraAuthGateway.for_testing() generates self-signed test tokens internally.

The app registration manifest in Azure Portal must include "groupMembershipClaims": "SecurityGroup" for group OIDs to appear in tokens. Without this, claims.groups will be empty and data scope will always be [].
