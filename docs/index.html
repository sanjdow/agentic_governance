Data Access Governance for LLM Agents

SD
sanjdow
· May 2026 · 12 min read

Current approaches to governing agentic AI systems conflate identity authorization with query authorization. A token proves who is calling. It says nothing about whether the specific query being executed — generated dynamically at runtime by a language model — is policy-compliant against the data it touches.

This repository is a Python reference implementation of a distinct architectural model in which governance policy originates exclusively in the data catalog, is enforced at the orchestration layer through policy-driven agent selection, and is carried into every data source call as a cryptographically signed, operation-scoped proof of prior authorization. The MCP server becomes a pure access enforcer, never a decision point.

Core thesis

The fundamental error in today's agent access control architectures is that they grant access to a source, not to a query. A token proves identity; it does not prove that the specific query being executed is policy-compliant. True agent access control requires that every tool call carries a pre-validated, signed proof that the query itself — not merely the agent — has been authorized.

The model inverts the prevailing trust assumption. In token-based systems, the trust model is reactive: a token grants entry and any subsequent query is presumed acceptable unless the receiving system detects a violation. Here, it is proactive: no query reaches a data source unless it arrives with a signed attestation that it has already been found compliant.

Architecture — Four governance layers

Every request resolves a UserContext at §0 (Entra ID authentication), which propagates unchanged through L1–L4 as the root of all trust decisions. No layer can be bypassed.

§0
Auth
Entra ID → UserContext

Bearer token validated against JWKS. The Entra OID becomes user_id — immutable, unlike UPN. Group memberships map to data scope. App roles map to clearance level. OBO degradation applied explicitly when the token arrives via a delegation chain.

L1
Catalog
Data Catalog — authoritative source of access policy

Governance rules must not be defined in gateways, orchestrators, or MCP servers. They must be derived from the data catalog — the system of record for classifications, sensitivity levels, access rights, and regulatory constraints. Any policy change propagates outward. No downstream component holds an independent copy.

sensitivity classificationsaccess rightsPII columnsconsent statepolicy versioning
L2
Orch.
Orchestrator — policy at the agent selection layer

Before any agent is invoked, the orchestrator resolves which agents are eligible given the current user context, session, and request classification — as derived from catalog policies. Non-eligible agents are never activated. This is exclusion before invocation, not filtering after the fact.

clearance ceiling checkaccess right validationsource allowlistscope intersection
L3
Proof
Policy Resolver — authorization must precede the query

An agent expresses intent. The Policy Resolver validates it against catalog-derived access rights and issues an Signed Access Token: a cryptographically signed, short-lived document binding the specific query to the user identity, agent identity, session, and policy version at the time of issuance. The query hash is binding — any modification is detected. Each token is single-use.

SHA-256 query bindingRS256 JWT signing120s TTLnon-delegablesingle-use
L4
MCP
MCP server — access enforcer, never decision point

The MCP server does not evaluate whether a query is permissible. That decision has already been made and encoded in the proof. The server verifies the proof's signature, checks expiry, applies replay protection, and either pushes catalog-derived filters down to the data source or executes the pre-validated query unchanged.

signature verificationreplay protectionfilter push-downcolumn masking

Three additional controls apply across all layers: context governance (classifies inter-agent context and redacts above clearance ceiling), governed cache (identity-scoped Redis keys, policy-version TTLs, Fernet encryption at rest), and session isolation (agents declare exactly which fields they need via ContextRequest — no raw context forwarding).

Ten failure modes addressed

Each of these is a concrete, exploitable gap in standard agentic deployments. The framework addresses all ten with specific implementations in the repository.

02RBAC cannot govern non-deterministic queries. LLM agents generate queries at runtime. The same prompt can produce vastly different surfaces. RBAC cannot govern what it cannot predict.orchestrator/

05Context leaks across agent chains. Data retrieved under governed conditions exits the governance perimeter the moment it enters a context window. No policy follows it forward.context_governance/
06Redis cache carries no classification metadata. Keys rarely incorporate identity, enabling cross-user and cross-agent data leakage via cache hits.cache/governed_cache.py

08MCP server making access decisions inline. When access control logic lives inside the MCP server it is scattered across every data source, drifts inconsistently, and cannot scale.mcp_server/

The Signed Access Token

The core artifact of the data access governance model. A cryptographically signed JWT binding a specific query to a specific user, agent, session, and catalog policy version. Five properties make it tamper-evident and non-reusable.

JWT Payload Schema issued by Policy Resolver · RS256 · 120s TTL
jti
Unique proof identifier. Tracked by the MCP server for replay protection — a proof can only be used once.
sub
Entra OID of the invoking user. Immutable identity anchor — survives OBO delegation when embedded in the proof.
agent_id
The agent the proof was issued for. A different agent presenting this proof is rejected — non-delegable by design.
query_hash
SHA-256 of the canonicalized query. Any downstream query substitution is detected and rejected.
policy_version
Catalog policy checksum at time of issuance. Proofs issued under superseded policy versions are detectable.
allowed_filters
Catalog-derived row filters and column masks. Pushed down to the data source by the MCP enforcement server.
exp
Expiry timestamp. Default 120 seconds from issuance. No refresh — expired proofs are cryptographically unextendable.
# Request a proof from the Policy Resolver
proof = resolver.request_proof(
    session=session,           # contains UserContext with Entra OID
    agent=retrieval_agent,
    query="SELECT model, rate FROM quality.defect_rates",
    asset_ids=["quality_metrics"],
)

# Make a governed MCP call — proof travels with the query
result = mcp.handle_tool_call(MCPToolCall(
    tool="query_datasource",
    arguments={
        "source": "snowflake",
        "query": query,
        "agent_id": "retrieval_agent",
    },
    proof=proof,    # ungoverned calls rejected at L4
))

Entra ID integration

Entra ID provides the identity anchor. The Signed Access Token provides the authorization artifact. These solve different problems. The auth/ module bridges them by translating Entra claims into the framework's UserContext.

✓ What Entra ID provides
Principal authentication

Users, Managed Identities, and Service Principals authenticated via OAuth 2.0/OIDC. The OID is the immutable identity anchor — unlike UPN, it never changes on username renames.

Conditional Access pre-filter

Location, device compliance, and MFA enforcement can gate token issuance — a coarse pre-filter before any agent is invoked.

Group and role claims

Group OIDs map to data scope values; app role values map to clearance levels via configurable tables in EntraConfig.

✕ What Entra ID cannot prove
Query-level compliance

A valid Bearer token proves authentication — not that the specific query being executed is policy-compliant for the asset it touches.

Agent vs. user identity

When an LLM agent calls an MCP server using Managed Identity, the token belongs to the agent's service principal. The invoking user is invisible to the data source.

OBO delegation integrity

On-Behalf-Of propagates OID and UPN — but not app role assignments. Data scope and clearance level are silently lost in delegation chains.

The fix

Embed the Entra OID in the Signed Access Token as sub. Identity, data scope, clearance, and query hash travel together in a single signed artifact. Even when downstream agents never see the original Bearer token, the identity and policy context are preserved and verifiable. The apply_obo_constraints() method in auth/claim_mapper.py makes OBO degradation explicit — clearing brand_scope and reducing clearance_level to INTERNAL rather than silently inheriting stale context.

Module overview

auth/Entra ID authentication. Token validation against JWKS, claim mapping (groups → scope, roles → clearance), OBO constraint modelling, test token factory for credential-free testing.
catalog/L1 data catalog. In-memory stub with runtime-queryable APIs for classifications, access rights, consent state, row filters, and policy versioning. Replace with a real catalog (Databricks Unity Catalog, Microsoft Purview) in production.
policy_resolver/L3 Policy Resolver. Issues RS256-signed Signed Access Tokens. Handles per-proof and session-level revocation. Private key never leaves this module.
orchestrator/L2 agent eligibility. Resolves eligible agents against catalog policy before invocation. Blocks agents where the user's clearance is below the agent's max_sensitivity ceiling.
mcp_server/L4 MCP Governance Server. Pure enforcement point. Verifies signature, expiry, and replay. Routes to filter push-down or pass-through. Never decides policy.
context_governance/Inter-agent context middleware. Heuristic PII and sensitivity classifier with overlapping chunk windows to prevent boundary evasion. Redacts content above the receiving agent's clearance ceiling.
injection_detection/4-layer prompt injection detector: critical patterns, boundary analysis, tool-redirect detection, delimiter scanning. Applied before any retrieved content reaches an agent's context.
cache/Governed Redis wrapper. Identity-scoped keys (user + agent + query hash). Policy-version-bound TTLs. Fernet encryption at rest. Falls back to in-process store when Redis is unavailable.
agents/Session isolation. SessionStateStore with need-to-know ContextRequest. Agents declare which output types and field names they require — no raw context forwarding between hops.
core/Shared types. All models, exceptions, and identifier validation. SensitivityLevel comparisons. AuthorizedQueryProof schema. SQL-injection-safe identifier enforcement on UserContext and AgentContext.

Installation

Requires Python 3.10+. No external services or Azure credentials required to run the full test suite — the Entra ID integration uses a self-signed test token factory in development mode.

# Clone
git clone https://github.com/sanjdow/agentic_governance.git
cd agentic_governance

# Virtual environment
python3 -m venv .venv && source .venv/bin/activate  # macOS/Linux

# Install
pip install -r requirements.txt         # simple
pip install -e ".[test]"               # editable + tests
pip install -e ".[redis]"              # real Redis backend

# Run the test suite (80 tests)
pytest tests/ -v

# Core governance demo
python examples/full_scenario.py

# Entra ID auth demo (no Azure credentials needed)
python examples/demo_entra_auth.py
Entra ID production configuration

Set ENTRA_TENANT_ID, ENTRA_CLIENT_ID, and ENTRA_AUDIENCE environment variables. Add "groupMembershipClaims": "SecurityGroup" to your Azure app registration manifest so group OIDs appear in tokens. See the full README for the complete configuration reference.
