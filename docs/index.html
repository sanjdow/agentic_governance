The Core Problem

Tokens Prove Identity.
Not Query Compliance.

Traditional API governance assumes deterministic callers making predictable requests. LLM agents are neither. They generate SQL at runtime, chain calls across permission boundaries, and accumulate data in context windows that no policy engine can see.

✕  Traditional API Governance

Reactive — assumes determinism

A token grants entry. Any subsequent query is presumed acceptable unless the receiving system detects a violation. Policy logic must be replicated across every system.

  • ✕ Identity maps to roles — queries are unconstrained
  • ✕ Audit trail covers actions, not intent
  • ✕ No cross-call memory — aggregation attacks invisible
  • ✕ Policy logic scattered across every data source
  • ✕ Managed Identity tokens have no user context
✓  Pre-Authorization Model

Proactive — query must prove compliance

No query reaches a data source without a signed attestation it has already been found compliant. The query is the proof.

  • ✓ Catalog is the sole source of governance truth
  • ✓ Agents blocked before invocation, not filtered after
  • ✓ Signed, query-bound proof — non-delegable, single-use
  • ✓ MCP server enforces proofs — never decides policy
  • ✓ Entra OID embedded — identity survives OBO delegation
Architecture

Four Governance Layers

Every request flows through four enforced layers before a single byte of data moves. Each layer has a single responsibility and cannot be bypassed.

Step 0 · Auth
Entra ID Authentication
Bearer token validated against JWKS. OID → user_id. Group memberships → data scope. App roles → clearance level. OBO degradation applied explicitly.
RS256 JWT validation group → scope mapping OBO constraint modelling Managed Identity rejection
L1 · Policy Source
Data Catalog — Single Source of Governance Truth
All governance rules originate here. No downstream component holds its own copy of policy logic. Any change propagates outward automatically.
sensitivity classifications access rights data scope tags PII column lists consent state policy versioning
L2 · Eligibility
Orchestrator — Agent Gating Before Invocation
Before any agent runs, the orchestrator resolves which agents are eligible given catalog policies and user context. Non-eligible agents are never activated — this is exclusion before invocation, not filtering after.
clearance ceiling check access right validation source allowlist data scope intersection
L3 · Pre-Authorization
Policy Resolver — Authorized Query Proof
An agent expresses intent. The resolver validates the specific query against catalog policy and issues a cryptographically signed proof. The query hash is embedded — the query cannot be substituted downstream. No agent can self-issue.
SHA-256 query binding RS256 signing 120s TTL non-delegable single-use row filter derivation
L4 · Enforcement
MCP Governance Server — Enforcement Point Only
The MCP server verifies the proof's signature and expiry, then routes to filter push-down (SQL WHERE injection) or pass-through. It never makes a policy decision — that was already made and signed in L3.
signature verification replay protection filter push-down column masking
Cross-Cutting Controls
Context Governance
Classifies inter-agent context. Redacts content above the receiving agent's clearance ceiling before it passes.
Governed Cache
Identity-scoped Redis keys, policy-version TTLs, Fernet encryption at rest. Sensitive data never cached in plaintext.
Session Isolation
Agents declare exactly which fields they need. SessionStateStore mediates access — no raw context forwarding.
The Core Artifact

The Authorized Query Proof

A cryptographically signed JWT that binds a specific query to a specific user, agent, session, and catalog policy version — at the time of authorization. Five properties make it tamper-evident and non-reusable.

Proof Properties

  • query-bound SHA-256 hash of the exact query embedded. The query cannot be substituted at the MCP layer.
  • non-delegable agent_id embedded. A different agent presenting the same proof is rejected immediately.
  • single-use proof_id tracked in the MCP server. Replay attacks are blocked by design.
  • short-lived Default 120-second TTL. No refresh mechanism. Expired proofs are cryptographically unextendable.
  • policy-versioned Catalog policy version embedded. Proofs issued under superseded policies are detectable.

JWT Payload Schema

{
  "jti":            "proof-uuid",
  "sub":            "entra-oid-immutable",
  "agent_id":       "retrieval_agent",
  "session_id":     "session-uuid",
  "query_hash":     "sha256:a3f8...",
  "policy_version": "v:c4e2a1b9...",
  "allowed_filters": {
    "scope_filter": "scope IN ('x')"
  },
  "iat": 1716000000,
  "exp": 1716000120,
  "iss": "policy-resolver"
}
Failure Modes Addressed

10 Ways Governance Breaks
with LLM Agents

Each of these failure modes has a concrete implementation in the framework. Click through to the GitHub repo for the module implementing each fix.

01
Tokens prove identity, not query compliance
A valid Bearer token says who is calling. It says nothing about whether the specific SQL or API call being made is permitted against the data it touches.
→ policy_resolver/
02
RBAC cannot govern non-deterministic queries
LLM agents generate queries dynamically at runtime. The same user prompt can produce vastly different query surfaces. RBAC cannot govern what it cannot predict.
→ orchestrator/eligibility.py
03
Cross-call aggregation attacks
An agent can chain individually permitted calls across multiple data sources and correlate results in its context window into inferences no single source would have disclosed.
→ orchestrator/eligibility.py
04
Prompt injection hijacks tool calls
Malicious instructions embedded in retrieved data redirect agent behaviour. The agent makes tool calls under its own permitted identity — RBAC is completely blind.
→ injection_detection/
05
Context leaks across agent chains
Data retrieved under governed conditions immediately exits the governance perimeter when it enters an agent's context window. No policy follows the data forward.
→ context_governance/
06
Redis cache carries no classification
Cached tool results carry zero sensitivity metadata. Cache keys rarely incorporate identity, enabling cross-user and cross-agent data leakage through cache hits.
→ cache/governed_cache.py
07
Context window as shared memory
Without session isolation, Agent A's full context automatically becomes Agent B's context. Agents accumulate sensitive data outside any governed store.
→ agents/session_isolation.py
08
MCP server as policy decision point
When the MCP server makes policy decisions, policy logic must be replicated everywhere. It drifts, becomes inconsistent, and cannot scale across heterogeneous sources.
→ mcp_server/
09
Entra ID proves who, not what
A valid Entra Bearer token proves authentication. It does not prove that the specific query being executed is policy-compliant for the data it touches.
→ auth/
10
OBO delegation silently drops policy context
On-Behalf-Of propagates the user's OID — but loses app role assignments, data scope, and clearance level. Downstream agents operate with an invisible degraded context.
→ auth/claim_mapper.py
Identity Integration

Entra ID — Where Identity
Ends and Authorization Begins

Entra ID is the identity anchor. The Authorized Query Proof is the authorization artifact. These solve different problems — the auth/ module bridges them.

✓  What Entra ID provides
  • Principal authentication
    Users, Managed Identities, and Service Principals authenticated via OAuth 2.0/OIDC. The OID is the immutable, canonical identity anchor.
  • Conditional Access pre-filtering
    Location, device compliance, and MFA enforcement can gate whether a session token is issued — a coarse L2-style pre-filter before agents are invoked.
  • Token propagation to MCP
    An Entra Bearer token can be forwarded to MCP servers, establishing that a principal with a known OID made the call.
✕  What Entra ID cannot prove
  • Query-level compliance
    A valid Bearer token proves authentication — not that the specific query is policy-compliant for the data asset it touches.
  • Agent vs. user identity
    When an LLM agent calls an MCP server using Managed Identity, the token belongs to the agent's service principal. The invoking user is invisible to the data source.
  • OBO delegation chain integrity
    On-Behalf-Of propagates OID and UPN — but not app role assignments. Data scope and clearance level are silently lost in delegation chains.
The fix: embed the Entra OID inside the Authorized Query Proof — binding identity, session context, data scope, clearance, and query hash into a single signed, tamper-evident artifact that the MCP server enforces. Even when downstream components never see the original Bearer token, the identity is preserved and verifiable.
Implementation

Module Breakdown

Each module has a single, well-defined responsibility. See the GitHub repo for full implementation details, 80 tests, and runnable examples.

auth/
Entra ID Authentication
Token validation, claim mapping, OBO constraint modelling. Groups → data scope. Roles → clearance. Test factory for credential-free testing.
catalog/
Data Catalog
L1 policy source of truth. Runtime-queryable APIs for classifications, access rights, consent state, row filters, and policy versioning.
policy_resolver/
Policy Resolver
Issues RS256-signed Authorized Query Proofs. Short-lived, non-delegable, query-bound. Handles revocation and session-level invalidation.
orchestrator/
Agent Eligibility
Resolves eligible agents against catalog policy before invocation. Blocks agents exceeding user clearance. Prevents identity elevation attacks.
mcp_server/
MCP Governance Server
Pure Policy Enforcement Point. Verifies proof signature, expiry, and replay. Routes to filter push-down or pass-through. Never decides policy.
context_governance/
Context Governance
Heuristic PII and sensitivity classifier. Intercepts inter-agent context and redacts content above the receiving agent's clearance ceiling.
injection_detection/
Injection Detector
4-layer prompt injection detection before tool dispatch: critical patterns, boundary analysis, tool-redirect, delimiter injection scanning.
cache/
Governed Cache
Identity-scoped Redis wrapper. Policy-version-bound TTLs. Fernet encryption at rest. Prevents cross-identity and stale-policy cache hits.
agents/
Session Isolation
SessionStateStore with need-to-know ContextRequest. Agents declare exactly which fields they need — no raw context forwarding between hops.
Get Started

Up and Running in Minutes

No Azure credentials or external services required for development and testing. The full governance chain runs in-process.

  1. Clone the repository

    All source code, tests, and examples are in the sanjdow/agentic_governance repo.

  2. Create a virtual environment

    Python 3.10+ required. python3 -m venv .venv && source .venv/bin/activate

  3. Install dependencies

    pip install -r requirements.txt or pip install -e ".[test]" for an editable install.

  4. Run the test suite

    All 80 tests should pass. pytest tests/ -v

  5. Run the demos

    python examples/full_scenario.py for the core governance chain, or python examples/demo_entra_auth.py for the Entra ID flow.

# Clone
git clone https://github.com/sanjdow/agentic_governance
cd agentic_governance

# Install
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

# Verify — 80 tests pass
pytest tests/ -v

# Core governance demo
python examples/full_scenario.py

# Entra ID auth demo (no Azure creds needed)
python examples/demo_entra_auth.py
NOTE
Entra ID integration uses a test token factory in development mode — no Azure tenant, client ID, or credentials required to run all 80 tests.
